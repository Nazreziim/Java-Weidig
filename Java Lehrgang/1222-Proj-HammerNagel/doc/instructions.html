Anwendungsprogrammierer Java - Projektarbeit

Einführung und Inhaltsverzeichnis

Diese Dokument beschreibt, welche Dinge Sie benötigen und was Sie tun müssen, um Ihre Lösung für die Projektarbeit im Rahmen des Java-Lehrgangs an der ITSBw einzureichen. Bitte lesen Sie die Anweisung genau durch bevor Sie mit der Bearbeitung beginnen. Dieses Dokument enthält verbindliche Anweisungen, wie die Prokjektarbeit durchgeführt werden muss und gewährleistet auf diesem Weg Konsistenz und Fairness in der Bewertung.

Für die Bearbeitung der Aufgabe bekommen Sie folgende Quelldateien ausgehändigt:

  • Dieses Dokument

  • Eine nicht-relationale Datenbankdatei.

Stellen Sie sicher, dass Sie während der Bearbeitung eine Sicherheitskopie der Quelldateien aufbewahren, falls während Ihres Entwicklungsprozesses Fehler auftreten sollten. Es ist verboten für die Bearbeitung die Quelldateien anderer Lehrgangsteilnehmer zu nutzen (selbst wenn Sie der Meinung sind, dass diese über die selben Quelldateien verfügen).

Dieses Dokument ist wie folgt aufgebaut:

  • Überblick - Genereller Überblick über die zu entwickelnde Applikation

  • Architektur - Die Bestandteile der Applikation und wie sie zusammenarbeiten sollen

  • User Interface - GUI Spezifikation

  • Datenzugriff Vorgaben zur Umsetzung des Zugriffs auf die Datenbank-Datei

  • Datenformat Das Format und Schema der Datenbank-Datei

  • Abzuliefernde Artefakte - Beschreibung der Bestandteile, die Sie abliefern müssen und wie diese zu strukturieren sind

  • Bewertung - Wie wird Ihr Ergebnis bewertet?

  • Was soll ich tun, wenn ich eine Frage habe?

Wichtiger Hinweis: Automatisches Nichtbestehen

Wenn in diesem Dokument das Wort "müssen" benutzt wird, ist damit eine verbindliche Anforderung berschrieben! Sollten Sie eine solche Anforderung nicht befolgen, wird die Arbeit automatisch ohne weitere Überprüfung als nicht bestanden gewertet. In Ihrem eigenen Sinne ist es daher von oberster Wichtigkeit, absolute Aufmerksamkeit auf die Verwendung des Wortes "müssen" zu legen! Teile Ihres Ergebnis werden ggf. automatisch per Software überprüft - sollte also eine bestimmte Struktur oder Namensgebung vorgeschrieben sein, so könnte hier schon die kleinste Abweichung zu einem automischen Nichtbestehen führen. 

Überblick

Hintergrund

Hammer & Nagel GmbH ist ein Vermittlungsunternehmen für Handwerker. Das Unternehmen verarbeitet Anfragen von Immobilienbesitzern für einen speziellen Arbeitsauftrag und bietet dem Kunden jeweils einen oder mehrere Vertragsnehmer an, die den Auftrag erfüllen können. Aktuell stellt Hammer & Nagel GmbH diese Dienstleistung über ein Customer-Service-Team (CST) per Telefon zur Verfügung. Das CST arbeitet mit einer in die Jahre gekommenen, selbst entwickelten Applikation, die inzwischen deutlich kritisiert wird. Daher möchte Hammer & Nagel GmbH zukünftig auf webbasiertes Marketing setzen und seine Serviceleistung über das Internet direkt den Kunden verfügbar machen.

Der Chief Information Officer (CIO) der Firma hat entschieden, die existierende Applikation in eine Java-basiertes System zu migrieren. Für den Anfang soll das System nur das CST unterstützen, obwohl die Hoffnung besteht, dass dieser Zwischenschritt als Startpunkt für die Migration in eine webbasierte Anwendung genutzt werden kann. Trotz der Tatsache, dass sich der CIO nicht die höchste Wiederverwendbarkeit von dem neuen System erwartet, setzt er darauf, dass sich die Entwicklung einer Java-Anwendung als lehrreiche Vorübung für ein webbasiertes System erweist.

Die IT-Administration von Hammer & Nagel ist im Besitz einer Daten-Datei, welche die Datenbasis der Firma enthält. Da diese Daten weiterhin noch in einer anderen Applikation (Reporting) benutzt werden, muss die neue Applikation das Format der Daten-Datei ohne Abänderung übernehmen und nutzen können.

Die neue Anwendung, welche das ursprüngliche Datenformat beibehält, muss in der Lage sein für das CST eine Liste aller möglichen Auftragnehmer zu generieren, die den Kriterien des Kunden entsprechen. Für die Umsetzung dieses Projekts haben Sie den Zuschlag erhalten.

Was Sie tun müssen

Die folgenden "top level" Features müssen implementiert werden:

  • Eine Client-Applikation, die über ein User-Interface Zugriff auf die gespeicherten Datensätze ermöglicht.

  • Ein Zugriffssysten, welches das Sperren / Entsperren sowie die flexible Suche von Datensätzen ermöglicht.

Ihre Arbeit an dem Projekt wird eine Reihe von Design-Entscheidungen, die Sie zu treffen haben, mit sich bringen. Wenden Sie hierbei stets folgende Richtlinien an:

Übersichtlichkeit und Wartbarkeit

Ein klares Design, welches auch von Prgrammieranfängern verstanden werden kann, wird einer komplexen Lösung vorgezogen, selbst wenn die komplexe Lösung ein wenig effizienter sein sollte. Die Komplexität des Codes (z.B. Verschachtelungstiefe, Parameterübergaben, Klassen- / Schnittstellenanzahl) soll sich in einem vernünftigen Rahmen bewegen.

Dokumentation

Sowohl Dokumentation (sowohl im Code als auch außerhalb) als auch die Wahl von Methoden- / Variablennamen muss in englischer Sprache erfolgen. Der Code sollte von sich aus schon so sprechend wie möglich sein; schreiben Sie keine Kommentare, die nichts zum Verständnis beitragen. Komplexe und missverständliche Abschnitt oder Methoden sollen mit beschreibenden Kommentaren versehen sein. Weiterhin muss jedes public- Element einer Klasse / eines Interfaces sowie jede Klasse / Interface selbst eine JavaDoc-Dokumentation haben. Spätestens mit Abgabe des Ergebnisses muss die komplette JavaDoc-Doku mit dem "javadoc" Tool im HTML-Format generiert und abgegeben werden.
Sie müssen die grundlegende Funktion Ihres Produkts dokumentieren. Diese Dokumentation soll ausreichend sein, dass sich ein mit dem Zweck des Produkts vertrauter Nutzer mit der Funktionalität vertraut machen kann. Dieser Teil der Dokumentation muss in einem der folgenden Formate erstellt sein:

  • HTML

  • Plain text (mit Texteditor lesbar)

  • Innerhalb der Applikation als Hilfe

Korrektheit

Ihr Projekt muss den Anforderungen dieser Spezifkation entsprechen. Bei Abweichungen werden Ihnen Punkte abgezogen. Zusatzpunkte für Arbeit, die über die Anforderungen dieser Spezifikation hinaus geht, gibt es nicht.

Benutzung von Standard-Sprachelementen

Ziehen Sie Funktionalität, die von Java-Standardklassen zur Verfügung gestellt wird Ihrer eigenen Implementierung dieser Funktionalität vor - es sei denn Ihre Implementierung bietet einen fallspezifischen Vorteil gegenüber dem Standard (dieser sollte dann aber dokumentiert werden).

Architektur

Hauptkomponenten

Im ersten Schritt des Umbaus bei Nagel & Hammer (dieses Projekt!) soll eine standalone Applikation erstellt werden. Dabei gibt es zwei Hauptkomponenten: Das Datenmanagement und die GUI. Trotzdem soll die Anwendung erweiterbar implementiert werden, um in einem Folgeprojekt eine Netzwerkverbindung zwischen den beiden Hauptkomponenten zu ermöglichen (Client-Server-Applikation).

Standalone Applikation

Im Standalone-Modus laufen GUI und Datenbank in der selben VM. Dabei sind keine Netzwerk-Operationen oder Serialisierungen notwendig.

User Interface

Das Nutzerinterface der Applikation muss den folgenden Kriterien genügen:

  • Es müssen ausschließlich Elemente des JDK verwendet werden.

  • Der Benutzer muss in der Lage sein, sich alle Datensätze anzeigen zu lassen.

  • Der Benutzer muss sich alle Datensätze mit einem bestimmten Namen und / oder Einsatzort gefiltert anzeigen lassen können.

  • Der Benutzer muss einen neuen Datensatz hinzufügen können.

  • Der Benutzer muss einen ausgewählten Datensatz löschen können.

  • Die Datensätze müssen stets tabellarisch angezeigt werden.

  • Der Benutzer muss in der Lage sein, einen Datensatz zu buchen, wobei die Datenbankdatei entsprechend aktualisiert wird.

  • Das Programm muss es dem Benutzer ermöglichen, den Pfad der Datenbankdatei zu ändern und diese Konfiguration persistent in der Datei config.ini zu speichern. Das Programm muss diese Datei bei jedem Start auslesen um die letzte Konfiguration wieder zu laden und die entsprechenden Datensätze anzuzeigen.

Ihre Nutzeroberfläche sollte zukünftige Erweiterungen nicht unterbinden und ggf. gegen eine andere GUI ausgetauscht werden können.

Datenzugriff

Notwendiges Interface

Ihre Datenzugriffsklasse muss "Data.java" heißen und sich im Paket "db" befinden. Außerdem muss sie das folgende Interface implementieren: 

package db;


public interface DBAccess
{
  // Reads a record from the file. Returns an array where each
  // element is a record value.
  public String [] readRecord(long recNo)
    throws RecordNotFoundException;


  // Modifies the fields of a record. The new value for field n 
  // appears in data[n]. Throws SecurityException
  // if the record is locked with a cookie other than lockCookie.
  public void updateRecord(long recNo, String[] data, long lockCookie)
    throws RecordNotFoundException, SecurityException;


  // Deletes a record, making the record number and associated disk
  // storage available for reuse. 
  // Throws SecurityException if the record is locked with a cookie
  // other than lockCookie.
  public void deleteRecord(long recNo, long lockCookie)
    throws RecordNotFoundException, SecurityException;


  // Returns an array of record numbers that match the specified
  // criteria. Field n in the database file is described by
  // criteria[n]. A null value in criteria[n] matches any field
  // value. A non-null  value in criteria[n] matches any field
  // value that begins with criteria[n]. (For example, "Fred"
  // matches "Fred" or "Freddy".)
  public long[] findByCriteria(String[] criteria);


  // Creates a new record in the database (possibly reusing a
  // deleted entry). Inserts the given data, and returns the record
  // number of the new record.
  public long createRecord(String [] data)
    throws DuplicateKeyException;


  // Locks a record so that it can only be updated or deleted by this client.
  // Returned value is a cookie that must be used when the record is unlocked,
  // updated, or deleted. If the specified record is already locked by a different
  // client, the current thread gives up the CPU and consumes no CPU cycles until
  // the record is unlocked.
  public long lockRecord(long recNo)
    throws RecordNotFoundException;


  // Releases the lock on a record. Cookie must be the cookie
  // returned when the record was locked; otherwise throws SecurityException.
  public void unlock(long recNo, long cookie)
    throws SecurityException;


}

Alle nicht-implementierten Excpetions in diesem Interface müssen als Klassen im Paket db umgesetzt werden. Jede dieser Klassen muss einen Standardkonstruktur sowie einen Konstruktur mit String-Argument, welches als Beschreibung der Excpetion dient, besitzen.

Alle Methoden, die eine RecordNotFoundException werfen, sollen dies tun, wenn ein spezifischer Datensatz entweder nicht existiert oder in der Datenbank-Datei als "deleted" markiert ist.

Locking

Auch wenn in diesem ersten Projekt eine Standalone-Applikation im Vordergrund steht, muss Ihre Anwendung im Hinblick auf eine zukünftige Version in der Lage sein eine Locking- Funktionalität zur Verfügung zu stellen. Dies muss wie im obigen Interface spezifiziert umgesetzt werden.

Datenformat

Das Format der Datenbank-Datei ist wie folgt spezifiziert:

Dateianfang
4 Byte numerisch, magic cookie value. Identifikation der Datenbank-Datei
4 Byte numerisch, Gesamtlänge jedes Datenbankeintrags in Byte
2 Byte numerisch, Anzahl der Felder pro Datenbankeintrag

Abschnitt zu Beschreibung des DB-Schema.
Wiederholung für jedes Feld eines Datenbankeintrags:
2 Byte numerisch, Länge des Feldnamens in Byte
n Bytes (definiert durch den vorhergehenden Eintrag), Name des Feldes
2 Byte numerisch, Feldlänge in Byte
Ende des Wiederholungs-Blocks

Datenabschnitt
Wiederholung bis zum Ende der Datei:
1 Byte "deleted" flag, 0 = aktiver Datensatz, 1 = gelöschter Datensatz
Datensatz, enthält Felder wie in der Schemadefinition spezifiziert, keine Trennzeichen zwischen Feldern, maximale Feldlänge wie in der Schemadefinition angegeben.

Dateiende

Alle numerischen Werte, die im Header beschrieben sind nutzen das Format der Klassen DataInputStream und DataOutputStream. Alle Textwerte und alle Felder (die nur aus Text bestehen) enthalten ausschließlich 8bit codierte Zeichen(8bit US ASCII).

Abzuliefernde Artefakte

Zielplattform und Ausführung

Sie müssen während der Bearbeitung der kompletten Aufgabe auschließlich die Java Plattform verwenden. Das fertige Produkt muss bei der Abgabe in einem Produktivsystem, welches eine Java-Version verwendet, die nicht älter als 1,5 Jahre ist, lauffähig sein.

Sie dürfen jede beliebige IDE benutzen; stellen Sie dabei jedoch sicher, dass sämlticher Code, den Sie abgegeben von Ihnen selbst verfasst wurde.

Zur Abgabe muss sich das Produkt mit folgender Kommandozeile starten lassen:

java -jar <path_and_filename>

Dabei muss kein weiterer Parameter über die Kommandozeile übergeben werden. Sämtliche ggf. notwendigen Konfigurationseinstellungen müssen über die GUI einstellbar sein und beim Verlassen des Programms persistent gespeichert werden. Der Name einer solchen Konfigurationsdatei muss config.ini lauten und diese Datei muss sich im aktuellen Arbeitsverzeichnis befinden. Das Programm muss in sich lauffähig sein, es darf keine manuelle Editierung von Dateien durch den Prüfer im Vorfeld notwendig sein.

Paketierung der abzugebenden Artefakte

Alle Elemente, die Sie zur Prüfung einreichen wollen, müssen in einer ZIP-Datei verpackt sein. Benennen Sie diese ZIP-Datei mit dem Abgabedatum in der Form YYYY-MM-DD_.zip. Die ZIP-Datei muss folgendes Layout und Inhalt haben:

  • Die ausführbare JAR-Datei Ihres Programms. Name: runme.jar.

  • Die originale und unveränderte Datenbank-Datei, die Ihnen zu Beginn des Projekts übergeben wurde. Denken Sie daran, eine Kopie der Datei aufzubewahren! Die Bewertung erfolgt auf exakt dieser Datenbasis.

  • Ein Verzeichnis namens code, in welchem sich sämtlicher Sourcecode Ihres Projekts in unkompilierter Fassung befindet. Sie müssen hier Unterverzeichnisse erstellen, um die Paketstruktur abzubilden.

  • Eine Datei namens version.txt. In dieser Datei müssen Sie die exakte Version des von Ihnen verwendeten JDK sowie des Betriebssystems aufführen. Die Datei muss im ASCII Format geschrieben sein.

  • Ein Verzeichnis namens docs, welches folgende Elemente enthält:

    • Diese HTML-Datei

    • Ein Unterverzeichnis namens javadoc, in welchem sich die HTML/Javadoc Dokumentation aller von Ihnen abgegebenen Klassen / Interfaces befindet.

    • Eine Datei namenschoices.txt. Diese Datei muss im ASCII Format geschrieben sein und die von Ihnen getroffenen Design Entscheidungen erläutern. Sie müssen hierbei die Probleme, auf die Sie gestossen sind, erläutern, Ihre Entscheidungsalternativen darstellen und Ihre getroffenen Entscheidungen begründen. Ebenfalls in diesem Dokument sollten Sie erlebte Unsicherheiten beschreiben, sowie die darauf folgenden Entscheidungen erklären. Bitte beachten Sie, dass es oftmals nicht den "einen richtigen Ansatz" gibt, sondern mehrere Wege zum Ziel führen können.

    • Die Benutzerdokumentation Ihres Programms wie im Abschnitt Überblick beschrieben. Falls Ihre Dokumentation im Programm selbst enthalten ist, können Sie diesen Punkt überspringen. In jedem anderen Fall müssen Sie entweder eine ASCII-Textdatei namens userguide.txt oder eine / mehrere verlinkte HTML-Dateien zur Verfügung stellen, die über userguide.html. gestartet werden können.

Bewertung

Wie funktioniert der Bewertungsprozess?

Die Bewertung erfolgt in drei Phasen: Zuerst wird überprüft, ob die Struktur Ihrer abgegebenen Datei formell der Spezifikation entspricht. Danach wird der Prüfer Ihre Applikation starten, um zu bewerten, ob die verlangten Funktionalitäten alle enthalten sind. Falls in diesen beiden ersten Phasen bereits Probleme auftreten, die nicht den "Muss"-Anforderungen dieser Spezifikation genügen, wird die Bewertung abgebrochen und das Projekt gilt automatisch als nicht bestanden.

Sollte das essenzielle Verhalten Ihrer Software korrekt sein, wird der Prüfer Details in Design und Umsetzung Ihrer Bearbeitung bewerten. Dieser Schritt ist deutlich zeitaufwendiger als die ersten beiden. Auch hier führt die Nichterfüllung von Muss-Anforderungen zum automatischen Nichtbestehen. Hier werden auch Ihre getroffenen Entscheidungen hinsichtlich unklarer Spezifikation bewertet. Solche Entscheidungen müssen in Ihren Kommentaren klar und knapp erläutert sein!

Bewertungskriterien

Ihre Arbeit wird anhand der folgenden Kriterien bewertet. Es gibt maximal 400 Punkte:

Grundsätzliche Überlegungen (80 Punkte)
Dokumentation (50 Punkte)
Objektorientiertes Design (50 Punkte)
GUI (70 Punkte)
Locking (80 Punkte)
Verwendetete Sprachmittel (70 Punkte)

Was soll ich tun, wenn ich eine Frage habe?

Sie werden möglicherweise an einigen Stellen dieses Dokuments eine Rückfrage haben, oder nach der Erlaubnis verlangen ein Problem speziell auf die eine oder andere Art lösen zu dürfen. Diese Spezifikation lässt absichtlich einige Dinge offen und erwähnt Problemstellen nicht explizit. Ihre Fähigkeit sich in eine solche Spezifikation einzuarbeiten entspricht der Realität und ist daher auch Teil des Bewertungsprozesses.

Generell gilt: Es gibt oft mehrere Wege zum selben Ziel. Fragen Sie erstmal nicht, sondern stellen Sie Ihre eigenen Überlegungen an und entwickeln Sie Optionen, die Sie ergreifen könnten. Beschreiben Sie diese in knappen Worten in Ihren Kommentaren und treffen Sie eine Entscheidung. Solange Sie keine Muss-Anforderung verletzen ist jede Entscheidung gleich gut, solange Sie sie begründen können. In sofern ist es auch nicht gewinnbringend Ihren Nachbarn zu fragen, denn im Zweifel müssen Sie Ihre Entscheidung selbst auch mündlich belegen können!

Für darüber hinausgehende Punkte steht Ihnen selbstverständlich Ihr Hörsaalleiter / Fachlehrer zur Verfügung.
